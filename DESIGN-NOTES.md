## 1. ออกแบบใหม่: "รายการ" ฝ่ายขาย — ขายเป็นต้น กางเป็นท่อน

### ที่มา
ฝ่ายขายเสนอราคาลูกค้าเป็น **"ต้น"** แต่การผลิต/คิดต้นทุน/คิดคิว ใช้หน่วย **"ท่อน"**
(เสาเข็ม 1 ต้น = ประกอบจาก 1/2/3/4… ท่อน)

### โมเดลที่ตกลง
**ระดับ "ต้น" (แถวหลัก):**
- dropdown: `ชนิด` / `ขนาด` / `เกรด` → เป็น "ตัวล็อก" ส่งต่อให้ทุกท่อน
- dropdown: `ยาว` / `หัวต่อ` → เป็น **ค่าตั้งต้น** ให้ท่อน
- `จำนวนท่อน/ต้น` → คุมว่าจะแตกกี่บรรทัดท่อนด้านล่าง
- `ระบบติดตั้ง`, `จำนวน (ต้น)`, `ค่าเสา/ต้น`, `ค่าตอก/ต้น` (revenue คิดต่อต้น)
- ❗ ไม่มี item code ที่ระดับนี้

**ระดับ "ท่อน" (กางใต้แถวต้น):**
- `ชนิด/ขนาด/เกรด` (เช่น C/60/A) **ล็อกตามต้น แก้ไม่ได้** (แสดงไอคอนกุญแจ)
- เหลือ dropdown แค่ `ยาว` + `หัวต่อ` (แก้รายท่อนได้ เช่น 12+12 หรือ 13+11)
- ระบบประกอบ `item number` ให้ (เช่น `C60A1200T02`)
- คิดพื้นที่หน้าตัด / ปริมาตร / ต้นทุน ที่ระดับนี้

### การคำนวณ
- คิว/ต้น = Σ ปริมาตรทุกท่อน
- คิวรวม = คิว/ต้น × จำนวนต้น
- ต้นทุนผูก key ด้วย `item number` ของท่อน (ใช้ `costByItem` ที่มีอยู่) → ท่อนรหัสซ้ำ = สเปค/ต้นทุนต่อหน่วยเดียวกัน
- โครงสร้าง: revenue ต่อต้น · ต้นทุนต่อท่อน

### การ parse item code (มีอยู่แล้วในโค้ด)
regex `^([A-Z]+)(\d+)([A-Z])(\d{4})` → ชนิด / ขนาด / เกรด / ความยาว(÷100)
- ส่วนหาง (เช่น `T02` = หัวต่อ) **ปัจจุบันยังไม่ถูกใช้** — จะเริ่มใช้ในงานนี้

### UI / ความหนาแน่น (ตกลงจาก mockup)
- เลย์เอาต์ master-detail: แถบ "ต้น" เป็นหัวสี + ท่อนเป็นบรรทัดย่อยชิดกัน (บีบแนวตั้ง — เผื่อ 1–10 ต้น × ≥3 ท่อน)
- header คอลัมน์ (ท่อน / ล็อก / ยาว / หัวต่อ / item number / คิว) อยู่ **ใต้แต่ละแถบต้น** (ไม่ใช่หัวตารางรวมอันเดียว)
- ยุบ/กาง รายต้นได้ + ปุ่ม "ยุบทั้งหมด / กางทั้งหมด"
- หัวข้อกลุ่มใช้คำว่า "รายการ"
- ใช้ธีม/สี/ฟอนต์เดียวกับ `index.html`
- 📄 หน้าตาเป้าหมาย (static mockup): [`mockups/sales-ton-thon.html`](mockups/sales-ton-thon.html) — เปิดในเบราว์เซอร์เพื่อดู

### ❓ คำถามที่ยังค้าง (ต้องตอบก่อนเริ่มเขียน)
`ยาว/หัวต่อ` ที่ระดับต้น เป็นแบบไหน:
1. **default** — ท่อนรับค่ามาแล้วแก้รายท่อนได้ (ยืดหยุ่น เผื่อ 13+11)
2. **ตายตัว** — ทุกท่อนเท่ากันหมด แก้รายท่อนไม่ได้ (กรอกเร็ว พลาดยาก)

### แผนทำเป็นเฟส
- **เฟส 1:** โครง ต้น→ท่อน ใช้งานได้จริง — dropdown จำนวนท่อน + แตกบรรทัด + รวมคิวอัตโนมัติ (ยังพิมพ์ item code ท่อนเองได้) ← **เริ่มที่นี่**
- **เฟส 2:** cascading dropdown สร้าง item code จากแคตตาล็อกใน Sheet (ต้องเพิ่มแคตตาล็อก ชนิด/ขนาด/ความยาว/หัวต่อ ใน `refData` ผ่าน `loadPileCosts` ก่อน)
- **เฟส 3:** auto-breakdown (BOM) เลือกต้นแล้วกางท่อนมาตรฐานให้เอง แก้เป็น exception ได้

## 2. เช็คลิสต์ก่อนใช้ข้อมูลจริง (ตอนนี้เป็น dummy)

ทำก่อนสลับ dummy → ข้อมูลจริง:
1. เปลี่ยน repo เป็น **private** (ตอนนี้ public — source เปิดเผยหมด รวมถึง API_URL และรหัสผ่าน)
2. เปลี่ยนรหัสผ่านทุกตัว: รหัสแผนกใน `DEPARTMENTS` + `ADMIN_PASSWORD` (หมายเหตุ: เป็นการเช็คฝั่ง client เท่านั้น ไม่ใช่ security จริง)
3. ใส่ shared-secret token ระหว่าง HTML ↔ Apps Script (แก้ทั้ง 2 ฝั่ง) กัน API_URL ถูกยิงตรง
4. ล้างข้อมูล dummy ใน `seed()`
5. (ถ้าต้องการ) สำรองโค้ด `.gs` เข้า repo ผ่าน clasp

## 3. การทำงานข้ามเครื่อง (seamless)

หลักการ: **อะไรสำคัญ = ต้องอยู่ใน git** (session/บทสนทนา/memory อัตโนมัติของ Claude Code เก็บใน local แต่ละเครื่อง ไม่ sync ข้ามเครื่อง)

**ทุกวัน:**
- เริ่มงาน → `git pull` ก่อนเสมอ
- เลิก/สลับเครื่อง → `git commit` + `git push` ให้ครบก่อนลุกจากเครื่อง

**ตั้งเครื่องใหม่:**
1. ติดตั้ง Claude Code + login บัญชีเดิม (oatinbox@gmail.com)
2. `git clone https://github.com/OatPaco/project.git`
3. เชื่อม/authenticate MCP connectors ใหม่ (Google Sheets / Gmail ฯลฯ — token ไม่ตามไป)
4. เปิด Claude Code ที่โฟลเดอร์ `project` โดยตรง → จะอ่าน `CLAUDE.md` + `DESIGN-NOTES.md` อัตโนมัติ ได้บริบทครบ

หมายเหตุ: บทสนทนาเก่ากับ memory อัตโนมัติจะไม่ตามไป แต่ `CLAUDE.md` + ไฟล์นี้ครอบคลุมบริบทสำคัญแล้ว (อยากเก็บ transcript เพิ่ม ใช้ `/export` ในเครื่องเดิม)
